5 ## About clangmetatool
6 
7 When we first started writing clang tools, we realized that there is a
8 lot of life cycle management that we had to repeat. In some cases,
9 people advocate the usage of global variables to manage the life-cycle
10 of that data, but that makes code reuse across tools even harder.
11 
12 Additionally, we also learned that when writing a tool, it will be
13 beneficial if the code is split in two phases. First a data collection
14 phase, and later a post-processing phase that actually performed the
15 bulk of the logic of the tool.
16 
17 Essentially you will only need to write a class like:
18 
19 ```C++
20 class MyTool {
21 private:
22  SomeDataCollector collector1;
23  SomeOtherDataCollector collector2;
24 public:
25  MyTool(clang::CompilerInstance* ci, clang::ast_matchers::MatchFinder *f)
26  :collector1(ci, f), collector2(ci, f) {
27  // the individual collectors will register their callbacks in their
28  // constructor, the tool doesn't really need to do anything else here.
29  }
30  void postProcessing
31  (std::map<std::string, clang::tooling::Replacements> &replacementsMap) {
32  // use data from collector1 and collector2
33  // generate warnings and notices
34  // add replacements to replacementsMap
35  }
36 };
37 ```
38 
39 And then you can use the `clangmetatool::MetaToolFactory` combined
40 with the `clangmetatool::MetaTool` in your tool's main function:
41 
42 ```C++
43 int main(int argc, const char* argv[]) {
44  llvm::cl::OptionCategory MyToolCategory("my-tool options");
45  llvm::cl::extrahelp CommonHelp
46  (clang::tooling::CommonOptionsParser::HelpMessage);
47  clang::tooling::CommonOptionsParser
48  optionsParser(argc, argv, MyToolCategory);
49  clang::tooling::RefactoringTool tool(optionsParser.getCompilations(),
50  optionsParser.getSourcePathList());
51  clangmetatool::MetaToolFactory< clangmetatool::MetaTool<MyTool> >
52  raf(tool.getReplacements());
53  int r = tool.runAndSave(&raf);
54  return r;
55 }
56 ```
57 
58 One way in which our initial tools got hard to write and maintain was
59 by trying to perform analysis or even replacements during the
60 callbacks. It was not immediately obvious that this would lead to
61 hard-to-maintain code. After we switched to the two-phase approach, we
62 were able to reuse a lot more code across tools.
63

66 ## Infrastructure
67 
68 ### `clangmetatool::MetaToolFactory`
69 
70 This provides the boilerplate for a refactoring tool action, since you
71 need a factory that passes the replacementsMap in to the frontend
72 action class.
73 
74 ### `clangmetatool::MetaTool`
75 
76 This provides the boilerplate of a FrontendAction class that will
77 perform data gathering and then run a post-processing phase that may
78 do replacements. This simplifies the writing of the code into a
79 constructor that registers preprocessor callbacks or ast matchers and
80 a postprocessing phase.
81 
82 ### clangmetatool cmake module
83 
84 When building a clang tool you are expected to ship the builtin headers from the compiler with the tool, otherwise the tool will fail to find headers like stdarg.h. Clang expects to find the builtin headers relative to the absolute path of where the tool is installed. This cmake module will provide a function called `clangmetatool_install` which will handle all of that for you, example at [skeleton/CMakeLists.txt](skeleton/CMakeLists.txt).

91 ## Reusable data collection
92 
93 Another part of this library consists of a number of "Data
94 Collectors". Those will be in the `clangmetatool::collectors`
95 namespace.
96 
97 "Data Collector" is a "design pattern" for reusing code in clang
98 tools. It works by having a class that takes the CompilerInstance
99 object as well as the match finder to the constructor and registers
100 all required callbacks in order to collect the data later.
101 
102 The collector class will also have a "getData" method that will return
103 the pointer to a struct with the data. The "getData" method should
104 only be called in the 'post-processing' phase of the tool.
105 
106 ## Constant Propagation
107 
108 Another part of this consists of constant propagators to assist
109 with analysis. Those will be in the `clangmetatool::propagation`
110 namespace.
111 
112 More specifically, the current implementation provides propagation for the
113 follwing types so that variables may be queried for their true values anywhere
114 within the control-flow, so long as the value is deterministic:
115 
116 * Constant C-style string propagator, which propagates constant strings through
117 the control flow graph
118 
119 * Constant integer propagation which propagates integer values through the code
120 considering references, pointers, const-ness of `int` & `int`-like types
121 
122 This could be useful for various purposes but especially for identifing
123 things like which database a function is actually calling out to, etc.
124 
125 ### `clangmetatool::propagation::ConstantCStringPropagator`
126 
127 This provides infrastructure (utilizing
128 `clangmetatool::propagation::ConstantPropagator` and
129 `clangmetatool::propagation::PropagationVisitor`) to propagate constant
130 C-style string values over the program. Resulting in the true value of a
131 variable wherever the value is deterministic and "<UNRESOLVED>" anywhere else.
132 
133 #### `clangmetatool::propagation::ConstantPropagator` and `clangmetatool::propagation::PropagationVisitor`
134 
135 These two classes provide the boilerplate to create infrastructure
136 to propagate constants of arbitrary types through the control flow graph
137 of the program in such a way that anywhere the constant value of a variable
138 would be deterministic one may query its value at that point.
139 
140 These classes are private to the library, but additional propagators could be easily
141 made using these facilities.

143 ## Skeleton for a new project
144 
145 After you "git init" into an empty directory, copy the contents of the
146 skeleton directory. To build that project, do something like:
147 
148 ````bash
149 ( mkdir -p build && cd build/ && \
150  cmake \
151  -DClang_DIR=/path/to/clang/ \
152  -Dclangmetatool_DIR=/path/to/clang/ \
153  -DCMAKE_BUILD_TYPE=Debug \
154  -DCMAKE_EXPORT_COMPILE_COMMANDS=1 \
155  .. )
156 make -C build
157 ````

159 ## Building
160 
161 You need a full llvm+clang installation directory. Unfortunately, the
162 Debian and Ubuntu packages [are
163 broken](https://bugs.debian.org/cgi-bin/bugreport.cgi?bug=900440), so
164 you may need to work-around by creating some symlinks (see
165 .travis.Dockerfile in this repo for an example).
166 
167 ````bash
168 mkdir build
169 cd build
170 cmake -DClang_DIR=/path/to/clang/cmake ..
171 make
172 make install
173 ````
